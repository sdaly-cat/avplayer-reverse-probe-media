# AVPlayer Streaming-Reverse Probe — Handoff Brief

**Pick this up in a fresh Claude Code / VS Code session.** This repo is a self-contained native-iOS
sandbox; everything a new session needs is here. This is a **feasibility smoke test, not production
code.**

---

## The one question this probe answers

We're evaluating replacing the iOS/iPad coaching-player's reverse-playback machinery (a JS/WebCodecs
frame cache in the web app) with a native **`AVPlayer`** video surface. Native `AVPlayer` supports a
genuine negative `rate` (reverse), which HTML5 `<video>` in a WKWebView cannot do.

Everything hinges on one empirical unknown:

> **Does `AVPlayer` play our media *smoothly in reverse* while STREAMING a progressive MP4 over a
> CDN, at full res, on a real iPad — or is it only smooth for LOCALLY downloaded media?**

What we already know (from the sister native app Thunder Cloud + user reports):
- **Local/downloaded** media reverses fine.
- **Streaming over the network (range requests)** is where the performance complaints come from.

So the probe compares **streaming vs local** reverse on the **same clip**, on a **real iPad** (NOT
the Simulator — the Simulator uses the Mac's decoders and is not representative of iPad hardware
decode).

## The test media (hosted in this repo's Releases, tag `v1`)

Two **synthetic** clips, purpose-built to match our media's format. Identical except GOP structure:

| | |
|---|---|
| Resolution / fps | **1920×1080 / 59.94** |
| Codec | H.264 **High@4.2**, progressive (`frame_mbs_only_flag=1` — no iOS interlace trap) |
| GOP | 30, **closed** in one file / **open** in the other |
| Bitrate / size / len | 12 Mbps CBR / ~270 MB / 180 s |
| Streaming | faststart (`moov` at front), served over Fastly, honors HTTP Range, no auth/expiry |

- Closed GOP: `.../releases/download/v1/synthetic_ball_1080p_5994fps_3min_closedgop.mp4`
- Open GOP:   `.../releases/download/v1/synthetic_ball_1080p_5994fps_3min_opengop.mp4`

**Why two:** our real media is **open-GOP** (that's what tripped up WebCodecs on the web). Open GOP
is the harder, more representative case. Test both and compare. Swap `remoteURL` in `ProbeView.swift`
between `ProbeView.closedGOP` and `ProbeView.openGOP`.

**Burned-in overlays** (so the file is self-describing and reverse jank is visible):
- static format label (says `CLOSED GOP-30` / `OPEN GOP-30`)
- live timecode
- live **FRAME counter** (cyan) — in reverse, watch it decrement; smooth = good, stutter/skips = jank
- a DVD-style bouncing ball over a faint static grid (motion reference)

The clips were generated locally with ffmpeg (bouncing ball + `drawtext`); see the original build
notes in `cs-voltron-web/scratch/avplayer-reverse-probe-brief.md` if you need to re-encode.

## Two workflows to test (this is the important part)

- **Cold-region reverse:** scrub/seek to a spot that hasn't buffered yet, then hit `-1.0`. Worst
  case — network-bound. (The ~270 MB clip is big enough that seeks land in genuinely unbuffered
  regions.)
- **Watch-then-rewind (the real coaching workflow):** play forward ~10–15 s, pause, then hit `-1.0`
  to reverse back over the *just-watched* (already-buffered) region. **This is the case that
  actually matters** — coaches rewind what they just watched.

If watch-then-rewind is smooth but cold-region isn't, that says a compressed-range cache (à la
Thunder Cloud's `CacheAVAssetLoaderDelegate`) makes streaming reverse viable for the workflow we
care about.

Also record the three `canPlay*Reverse` flags (shown on screen) for both STREAM and LOCAL — a strong
signal on their own. They can be `false` until buffered; re-read after playing forward a bit.

---

## What the result means (report GREEN / AMBER / RED)

- **GREEN** — streaming reverse (esp. watch-then-rewind) is as smooth as local, full res →
  `AVPlayer` is viable for streaming; no compressed cache strictly required.
- **AMBER** — local smooth, streaming janky cold but smooth over the already-buffered region →
  viable **with** a compressed-range cache (port Thunder Cloud's `CacheAVAssetLoaderDelegate` idea).
  Most likely outcome.
- **RED** — reverse is janky even for local/downloaded media on a real iPad → native negative-rate
  is not the answer for our media; stop and reconsider.

---

## Gotchas

- **Real iPad only.** The Simulator decodes on the Mac and does not predict iPad reverse smoothness.
- **`canPlayReverse` may be `false` until buffered.** Give it a second; re-read after playing forward.
- **`VideoPlayer` transport controls can reset rate to 1** on play/pause — drive reverse with the
  on-screen rate buttons (they set `player.rate` directly). If the built-ins fight you, swap
  `VideoPlayer` for `AVPlayerViewController(showsPlaybackControls:false)` / a bare `AVPlayerLayer`.
- **Audio in reverse** will be absent/garbled — normal, ignore it; judge video only.

## What these synthetic clips do NOT replicate (next step for a representative test)

- **Real high-motion footage** (this is a flat bouncing ball) and a **real CloudFront edge with
  signed URLs**. To test that, get a real **progressive-MP4 rendition** from staging: the
  `cs-voltron-web` repo has a **`federation-crawl`** skill that dumps clips + VideoAsset renditions
  (with URIs) to JSON — pick a top-bitrate **progressive** (NOT interlaced, see VW-672) rendition and
  drop its signed URL into `remoteURL`. Signed URLs expire (mint a fresh one if it 401s).

## Bigger context (FYI)

This is the cheapest kill-switch test ("Q2") of a larger iOS AVPlayer feasibility spike
(`docs/plans/2026-07-14-ios-avplayer-reverse-feasibility-spike.md` in `cs-voltron-web`). The two
*harder* unknowns come later and are NOT part of this probe: (1) compositing a native `AVPlayerLayer`
under a transparent WKWebView with web-drawn annotations staying registered, and (2) driving native
player time + multi-angle sync across the JS↔native bridge. Thunder Cloud solved reverse natively but
drew/synced natively too, so it de-risks the *decoder*, not our web-hybrid. This probe only settles
the decoder/streaming-reverse question.
